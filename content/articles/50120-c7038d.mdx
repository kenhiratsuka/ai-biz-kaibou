# 不動産仲介業向け物件査定レポート・売却戦略提案書作成代行

## なぜ今なのか

日本の不動産仲介市場は、国土交通省の統計によると宅地建物取引業者数が約12万社（2023年度末時点）に達しており、そのうち9割以上が従業員10名以下の中小零細業者です。この「数は多いが人手が薄い」という構造が、AI自動化の最大の商機を生み出しています。

問題の核心は「査定レポートと売却戦略提案書」の作成工数にあります。不動産仲介業者が売主から媒介を獲得するためには、物件査定額の根拠を示す査定書と、「いつ・いくらで・どのように売るか」を示す売却戦略提案書をセットで提出するのが業界慣行です。ところがこれらの作成には、熟練スタッフでも1件あたり3〜5時間かかります。

大手不動産会社（三井のリハウス、住友不動産販売など）はこうした書類を社内テンプレートと専属スタッフで量産できますが、地域密着型の中小業者は対応が追いつかない。その結果、「査定依頼は来るが提案書が間に合わない」「テンプレートが古くて見栄えが悪い」「データ収集に時間がかかりすぎる」という悩みが慢性化しています。

一方、AI技術の進歩により、こうした構造化ドキュメントの作成は劇的に効率化できるようになりました。Claude 3.5 SonnetやGPT-4oは、物件情報・周辺相場データ・市場トレンドを入力すれば、プロ品質の文章を数分で生成できます。国内外のフリーランス市場でもこの種のサービスへの需要はじわじわと増えており、クラウドワークスやランサーズでは「不動産 資料作成」カテゴリの案件単価が1件1万5,000円〜3万円台で推移しています（2024年後半の実態相場）。

さらに、2024年から段階的に施行される不動産流通の電子化推進（電子契約の普及）により、書類の「見た目のクオリティ」が以前にも増して重視されるようになっています。デジタルで送付される資料は比較が容易なため、提案書の質が媒介獲得率に直結するようになってきました。この環境変化が、外部への作成代行ニーズをさらに押し上げています。

## AIで自動化できる部分の設計

このサービスの核心は、「情報収集→構造化→文章生成→フォーマット整形」という一連のフローをできる限り自動化し、人間の関与を最小限にすることです。以下のワークフローを設計します。

**全体フロー図（概念）**

```
クライアント（不動産業者）からの入力フォーム
        ↓
Google Formsで物件情報・周辺情報を収集
        ↓
Make.com（旧Integromat）がデータを整形・整理
        ↓
REINS・国土交通省の不動産取引価格情報（スクレイピング or API）で相場データ取得
        ↓
Claude API（claude-3-5-sonnet）へプロンプトとともに送信
        ↓
査定レポート本文・売却戦略提案書本文を生成
        ↓
Google Docs / Notion APIでドキュメントに自動整形
        ↓
PDFに変換してクライアントにメール or Slack通知
```

**各ツールの役割と設定のポイント**

- **Google Forms**：物件情報（所在地・面積・築年数・リフォーム履歴・現況など）と売主の希望条件（希望売却額・売却希望時期・売却理由）を構造化して収集。回答は自動的にGoogle Spreadsheetsに蓄積される。

- **Make.com（旧Integromat）**：Googleスプレッドシートの新規行追加をトリガーに、データをJSON形式に変換してClaude APIへ送信。月額9ドル〜のCore プランで十分に動作します。

- **Claude API（Anthropic）**：claude-3-5-sonnetモデルを使用。Input tokenあたり$0.003、Output tokenあたり$0.015という料金体系で、1件あたりの生成コストは約50〜80円程度に収まります。プロンプト設計が品質の8割を決めるため、ここに最も時間を投資してください。

- **国土交通省 不動産取引価格情報**：APIが公開されており（https://www.land.mlit.go.jp/webland/api.html）、無料で周辺の成約価格データを取得できます。Make.comのHTTPモジュールからGETリクエストを送るだけで連携可能です。

- **Google Docs API + Carbone.io**：テンプレートに自動で本文を流し込み、PDF化します。Carbone.ioは月額無料枠（100件/月）があり、副業初期はコストゼロで運用できます。

## 実際のツール設定（ステップバイステップ）

ここでは中核となるMake.com × Claude API連携の設定手順を詳細に説明します。

**ステップ1：Anthropic APIキーの取得**
1. https://console.anthropic.com にアクセスしてアカウント作成
2. 「API Keys」メニューから新規キーを生成（sk-ant-〜〜〜で始まる文字列）
3. 初期クレジット$5が付与されているため、テスト期間はコストゼロ

**ステップ2：Make.comアカウントとシナリオ作成**
1. https://www.make.com に登録（無料プランで1,000オペレーション/月）
2. 「Create a new scenario」をクリック
3. 最初のモジュールに「Google Sheets - Watch New Rows」を選択
4. Googleアカウントを連携し、物件情報を蓄積しているスプレッドシートとシートを指定

**ステップ3：HTTPモジュールでClaude APIを呼び出す**
1. 2つ目のモジュールに「HTTP - Make a Request」を追加
2. URL：`https://api.anthropic.com/v1/messages`
3. Method：POST
4. Headersに以下を追加：
   - `x-api-key`：取得したAPIキー
   - `anthropic-version`：`2023-06-01`
   - `content-type`：`application/json`
5. Body（JSON）に以下を設定（物件データは前ステップの変数を参照）：

```json
{
  "model": "claude-3-5-sonnet-20241022",
  "max_tokens": 4096,
  "messages": [
    {
      "role": "user",
      "content": "あなたは不動産査定の専門家です。以下の物件情報と相場データをもとに、売主向けの査定レポートと売却戦略提案書を作成してください。\n\n物件情報：{{物件情報の変数}}\n周辺成約事例：{{国交省APIの取得データ}}\n\n出力フォーマット：\n1. 査定額とその根拠（比較事例法）\n2. 売却推奨価格レンジ\n3. 売却戦略（価格設定・売り出し時期・販売方法）\n4. 想定スケジュール\n5. 注意点・付加価値提案"
    }
  ]
}
```

**ステップ4：国土交通省APIからの相場データ取得モジュール追加**
1. ステップ3の前に「HTTP - Make a Request」モジュールを追加
2. URL：`https://www.land.mlit.go.jp/webland/api/TradeListSearch?from=20241&to=20244&area={{都道府県コード}}&city={{市区町村コード}}`
3. レスポンスJSONから近隣成約事例を抽出し、Claude APIへのプロンプトに組み込む

**ステップ5：Google Docsへの自動書き込み**
1. 最後のモジュールに「Google Docs - Create a Document from a Template」を追加
2. 事前に作成したテンプレートドキュメントを指定
3. Claude APIのレスポンス（生成テキスト）をテンプレートの対応箇所に流し込む
4. 完成後、「Gmail - Send an Email」モジュールでクライアントに通知

**ステップ6：テスト実行と精度確認**
1. シナリオを保存後、「Run Once」でテスト実行
2. 実在の案件データ（ダミーOK）で出力品質を確認
3. プロンプトを調整し、5〜10件テストして品質を安定させる

## 自動化できない部分（ここが重要）

AI自動化で処理できるのは全工数の約60〜70%です。残りの部分は人間が必ず介在する必要があり、ここがサービスの差別化ポイントになります。

**1. 物件の個別特性の読み取りと補正**
AIは数値データは処理できますが、「日照が悪い南向き隣接ビルがある」「前面道路の交通騒音が激しい」「心理的瑕疵の近隣情報」といった定性情報を正確に査定に反映させるには、人間の判断が必要です。クライアントへの電話ヒアリングやGoogle Mapsでの目視確認は省略できません。

**2. 相場データの解釈と補正係数の設定**
国土交通省のデータは半期ごとの更新で、直近の市場変動（金利上昇・需給変化）を反映していません。REINSのリアルタイムデータは宅建業者しかアクセスできないため、最新の市況感は不動産業者との対話から取得し、AIの出力に手動で補正をかける必要があります。

**3. クライアントとの信頼関係構築**
不動産査定は売主にとって人生最大規模の意思決定に関わる資料です。「AI生成」とわかると信頼性を疑うクライアントも多いため、最終納品物はあくまで「専門家が作成した資料」として品質管理を人間が行う必要があります。

**4. 法的・倫理的チェック**
不動産広告や提案書には宅地建物取引業法上の制約（誇大広告の禁止等）があります。AIが生成した表現が法的にグレーゾーンにならないか、人間が最終チェックを行うことが必須です。

**5. 個別の価格交渉シナリオ設計**
「買取業者への売却 vs 仲介売却」「建物解体後の土地売却」などの比較シナリオは、地域の買取業者情報や地主・開発業者とのネットワーク情報が必要で、AIだけでは設計できません。

## 参入のボトルネック

正直に書きます。簡単ではありません。

**1. 不動産知識の学習コスト（最大の壁）**
査定レポートの「品質管理者」として機能するためには、比較事例法・原価法・収益還元法の基礎、路線価の読み方、REINSの仕組みなど、不動産の基礎知識が必要です。これを習得するのに最低1〜2ヶ月はかかります。逆に言えば、ここを乗り越えると参入障壁が高い分、競合も少ない。

**2. プロンプトエンジニアリングの試行錯誤**
Claude APIで「実際に不動産業者が使えるレベル」の提案書を生成するプロンプトを設計するには、20〜30回の試行錯誤が現実的に必要です。最初の数件は赤字覚悟でテスト案件として対応する気概が必要です。

**3. 初期クライアント獲得の難しさ**
不動産業者は保守的な業種であり、「外部の人間に査定業務を任せる」ことへの心理的抵抗が強い。最初の1社を獲得するまでが最も時間がかかります。無料モニター案件（2〜3件無償提供）で実績を作る戦略が有効ですが、その期間は収益ゼロが続きます。

**4. 法的グレーゾーンへの対応**
宅地建物取引業は免許が必要ですが、書類「作成代行」は免許不要です。ただし、作成した書類の内容に虚偽があった場合の責任の所在が曖昧になることがあります。クライアントとの契約書に「最終責任は発注者（宅建業者）にある」旨を明記する条項を必ず入れること。これを怠ると後でトラブルになるリスクがあります。

**5. Make.comとAPIの初期設定の技術ハードル**
プログラミング経験がない場合、前述のステップ3〜5の設定に詰まる可能性があります。Make.comのコミュニティフォーラム（英語）やYouTubeのチュートリアルで解決できることがほとんどですが、初期設定に週末1〜2回分の時間は見ておいてください。
